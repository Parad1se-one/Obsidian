## 🏗️ 核心架构设计

### 1. 智能体优先的代码仓库

```
初始状态 (2025.08):
├── 空仓库
├── 无预存人工代码
└── AGENTS.md 由 Codex 自行编写

5 个月后:
├── 应用逻辑 (Codex 生成)
├── 基础设施 (Codex 生成)
├── 工具链 (Codex 生成)
├── 文档 (Codex 生成)
└── 内部开发者工具 (Codex 生成)
```

### 2. 知识库设计 (关键创新)

**❌ 失败尝试**：单一大型 `AGENTS.md` (1000 页说明书)
- 情境稀缺，挤占任务空间
- 过多指导反而无效
- 迅速腐烂，难以维护
- 难以机械核实

**✅ 成功方案**：结构化 `docs/` 目录 + 轻量 `AGENTS.md` (约 100 行)

```
AGENTS.md (地图/目录)
    ↓
docs/ (结构化知识库)
├── 设计文档 (编目 + 索引 + 验证状态)
├── 架构文档 (域和包分层地图)
├── 执行计划 (进度 + 决策日志)
└── 技术债务清单 (版本控制)
```

**关键原则**：渐进式披露
- 智能体从小而稳定的切入点开始
- 被指导下一步该去哪里查看
- 避免一开始就被信息淹没

## 🔧 工程实践

### 1. 情境管理

**核心理念**：给 Codex 一张地图，而不是一本 1000 页的说明书

**实践方法**：
- 代码仓库内的知识 = 智能体能看到的全部
- Slack 讨论、Google Docs、人脑中的知识 = **对智能体不存在**
- 将情境推送到仓库中 (版本化、可检索)

**示例**：
```
❌ Slack 架构讨论 → 智能体无法发现
✅ 架构决策记录 (ADR) → 提交到仓库 → 智能体可检索
```

### 2. 架构约束

**严格分层模型** (每个业务域)：

```
Types → Config → Repo → Service → Runtime → UI
    ↓         ↓        ↓         ↓         ↓
  只能"向前"依赖，禁止循环
```

**横切关注点** (认证、遥测、功能标志)：
- 通过单一显式接口 `Providers` 进入
- 其他任何方式均被禁止

**强制执行**：
- 自定义 linter (Codex 生成)
- 结构测试
- CI 作业验证

### 3. 可观测性集成

**创新做法**：让应用程序对 Codex 直接可读

| 能力 | 实现方式 |
|------|---------|
| UI 测试 | Chrome DevTools 协议 + DOM 快照 + 截图 |
| 日志查询 | LogQL (智能体可直接使用) |
| 指标查询 | PromQL (智能体可直接使用) |
| 临时环境 | 每个 git worktree 启动独立实例 |

**效果**：
- Codex 可复现错误、验证修复
- 提示示例：`确保服务启动在 800ms 内完成`
- 提示示例：`这四个关键用户旅程中的任何跨度都不得超过两秒`

### 4. 自主智能体工作流

**端到端新功能开发** (Codex 可独立完成)：

```
1. 验证代码库当前状态
2. 复现已报告的漏洞
3. 录制演示故障的视频
4. 实施修复措施
5. 运行应用验证修复
6. 录制第二个视频 (演示解决方案)
7. 打开 Pull Request
8. 回应智能体和人类反馈
9. 检测并修复构建故障
10. 仅在需要判断时交人工处理
11. 合并更改
```

**运行时长**：单次任务可持续工作 **6+ 小时** (常在人类睡眠时间)

## 🧹 技术债务管理

### 问题：AI 残渣 (AI Residue)

**现象**：Codex 复现代码库中已存在的模式 (包括不良模式)
**初期方案**：每周五人工清理 (占 20% 时间) → **不可扩展**

### 解决方案：黄金原则 + 循环清理

**黄金原则** (编码到代码仓库)：
1. 优先使用共享实用程序包，而非手工辅助工具
2. 禁止"YOLO 式"探测数据 → 验证边界或使用类型化 SDK

**循环清理流程**：
```
后台 Codex 任务 (定期运行)
    ↓
扫描偏差 + 更新质量等级
    ↓
发起针对性重构 PR
    ↓
1 分钟内审查 + 自动合并
```

**类比**：垃圾回收 (Garbage Collection)
- 技术债务 = 高息贷款
- 小额持续偿还 > 累积后一次性痛苦解决

## 📈 工程规范变革

### 传统规范 vs 智能体优先

| 维度 | 传统规范 | 智能体优先 |
|------|---------|-----------|
| 合并门 | 严格阻塞 | 最小化阻塞 |
| 测试失败 | 无限期阻碍 | 后续重跑解决 |
| 纠错成本 | 高 | 低 |
| 等待成本 | 低 | **高** |
| PR 生命周期 | 长 | **短** |

**核心理念**：在智能体吞吐量远超人类注意力的系统中，**纠错成本低，等待成本高**

## 🎯 人类工程师的新角色

### 工作重心转移

```
传统工程师：
├── 编写代码 (70%)
├── 代码审查 (20%)
└── 架构设计 (10%)

智能体优先工程师：
├── 设计环境 (30%)
├── 明确意图 (25%)
├── 构建反馈回路 (25%)
├── 处理用户反馈 (15%)
└── 编写代码 (0%)
```

### 深度优先工作方式

**任务拆解**：
1. 将大目标拆解为小构建模块 (设计、代码、评审、测试)
2. 提示智能体构建这些模块
3. 用模块解锁更复杂任务

**问题排查**：
- ❌ 不是"再努力一点"
- ✅ "究竟还需要什么能力？如何让这个能力对智能体清晰可读且可强制执行？"

### 品味编码 (Encoding Taste)

**人类品味的反馈路径**：
```
审查评论 → 文档更新
重构 PR → 工具编码
用户 Bug → 规则转化
```

**原则**：当文档不够完善时，将规则转化为代码

## 💡 关键洞察

### 1. 纪律的重新定义

> "构建软件仍然需要纪律，但纪律更多地体现在支撑结构上，而不是代码上。"

**传统纪律**：代码规范、命名约定、代码审查
**新纪律**：工具、抽象、反馈回路、控制系统

### 2. 情境即代码

> "存储在 Google Docs、聊天记录或人们头脑中的知识都无法被系统访问。"

**推论**：
- 代码仓库 = 智能体的全部世界
- 未版本化的知识 = 不存在
- 为新队友准备的引导材料 → 同样为智能体准备

### 3. 约束即赋能

> "对于编码智能体来说，[严格架构] 是一个早期的先决条件：有了约束，速度才不会下降，架构才不会漂移。"

**反直觉**：
- 传统观点：约束降低速度
- 智能体优先：约束提升速度 (避免漂移和返工)

### 4. 人类品味的持续性

> "人类的品味一旦被捕捉，就会持续应用于每一行代码。"

**机制**：
- 审查评论 → 文档/工具
- 重构 PR → 编码规则
- 黄金原则 → 自动清理

## 📌 小虾点评

### 值得借鉴的实践

1. **结构化知识库**：AGENTS.md 作为地图而非百科全书
2. **架构约束编码化**：自定义 linter + 结构测试
3. **可观测性集成**：让智能体直接查询日志/指标
4. **循环清理机制**：技术债务的"垃圾回收"

### 适用场景

| 场景 | 适用度 | 说明 |
|------|-------|------|
| 新项目 (从零开始) | ⭐⭐⭐⭐⭐ | 无历史包袱 |
| 小型工具/原型 | ⭐⭐⭐⭐ | 快速迭代 |
| 大型遗留系统 | ⭐⭐ | 需大量重构 |
| 高可靠性系统 | ⭐⭐⭐ | 需严格验证 |

### 潜在风险

- **架构漂移**：需持续维护和清理
- **人类 QA 瓶颈**：智能体吞吐远超人类注意力
- **模型依赖**：系统行为高度依赖模型能力

### 行动建议

1. **从 AGENTS.md 开始**：编写轻量地图 (100 行左右)
2. **建立 docs/ 结构**：设计文档、架构文档、执行计划
3. **引入自定义 linter**：强制执行架构约束
4. **设置循环清理**：定期扫描技术债务
