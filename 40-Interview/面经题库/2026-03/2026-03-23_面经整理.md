#### 实现方案

**1. 规划阶段 (Plan)**

```python
async def plan_task(user_request: str, context: dict) -> List[Step]:
    prompt = f"""
    任务：{user_request}
    可用工具：{list_tools()}
    约束条件：{context.get('constraints', [])}
    
    请将任务分解为可执行的步骤，每个步骤包含:
    1. 步骤描述
    2. 所需工具
    3. 前置依赖
    4. 预期输出
    
    格式：JSON 数组
    """
    
    plan = await llm.generate(prompt, response_format='json')
    return parse_plan(plan)
```

**2. 执行阶段 (Execute)**

```python
async def execute_step(step: Step) -> ExecutionResult:
    try:
        # 获取工具
        tool = tool_registry.get(step.tool_name)
        
        # 准备参数
        args = await prepare_arguments(step, context)
        
        # 执行 (带超时和重试)
        result = await execute_with_retry(
            tool.call, 
            args, 
            timeout=step.timeout,
            max_retries=3
        )
        
        # 记录执行日志
        log_execution(step, result)
        
        return ExecutionResult(success=True, data=result)
        
    except Exception as e:
        return ExecutionResult(success=False, error=str(e))
```

**3. 反思阶段 (Reflect)**

```python
async def reflect_on_result(
    original_request: str,
    plan: List[Step],
    execution_results: List[ExecutionResult]
) -> Reflection:
    prompt = f"""
    原始请求：{original_request}
    执行计划：{plan}
    执行结果：{execution_results}
    
    请评估:
    1. 结果是否满足用户需求？
    2. 哪些步骤执行得好？哪些有问题？
    3. 如果需要改进，应该如何调整策略？
    4. 是否需要重新规划或补充执行？
    
    输出 JSON 格式评估报告。
    """
    
    reflection = await llm.generate(prompt, response_format='json')
    return parse_reflection(reflection)
```

#### 闭环控制逻辑

```python
async def agent_loop(user_request: str, max_iterations: int = 5) -> Response:
    context = initialize_context()
    
    for iteration in range(max_iterations):
        # 1. 规划
        plan = await plan_task(user_request, context)
        
        # 2. 执行
        results = []
        for step in plan:
            result = await execute_step(step)
            results.append(result)
            context.update(step, result)
        
        # 3. 反思
        reflection = await reflect_on_result(user_request, plan, results)
        
        # 4. 判断是否完成
        if reflection.is_complete:
            return synthesize_response(results, reflection)
        
        # 5. 根据反思调整
        if reflection.needs_replan:
            user_request = reflection.revised_request
            continue
    
    # 达到最大迭代次数，返回当前最佳结果
    return create_fallback_response(context)
```

#### 淘天场景优化

1. **电商知识注入**: 在规划阶段加入商品、订单、物流等领域知识
2. **工具优先级**: 高频工具 (查订单、查物流) 优先调度
3. **结果缓存**: 相似请求直接复用历史执行结果
4. **人工介入**: 低置信度时转人工客服

#### 缓存策略设计

**1. 多级缓存架构**:
```
L1: 本地缓存 (Guava/Caffeine)
    • 热点用户会话 (1000 个)
    • 过期：5 分钟
    
L2: Redis 集群
    • 活跃用户会话 (10 万个)
    • 过期：30 分钟
    
L3: MySQL + 向量库
    • 全量数据
    • 永久存储
```

**2. 缓存更新策略**:

```python
class MemoryCache:
    def __init__(self):
        self.l1_cache = CaffeineCache(max_size=1000, ttl=300)
        self.redis = RedisCluster()
        self.db = MySQL()
    
    async def get(self, user_id: str, key: str):
        # L1 命中
        if value := self.l1_cache.get(f"{user_id}:{key}"):
            return value
        
        # L2 命中
        if value := await self.redis.get(f"agent:memory:{user_id}:{key}"):
            self.l1_cache.set(f"{user_id}:{key}", value)
            return value
        
        # L3 读取
        value = await self.db.query(...)
        if value:
            await self.redis.set(f"agent:memory:{user_id}:{key}", value, ex=1800)
            self.l1_cache.set(f"{user_id}:{key}", value)
        return value
    
    async def set(self, user_id: str, key: str, value: dict, level: str = "short"):
        # 写入 L1
        self.l1_cache.set(f"{user_id}:{key}", value)
        
        # 写入 L2
        ttl = {"working": 1800, "short": 86400, "long": None}[level]
        await self.redis.set(f"agent:memory:{user_id}:{key}", value, ex=ttl)
        
        # 写入 L3 (异步)
        asyncio.create_task(self.db.upsert(user_id, key, value))
```

**3. 缓存淘汰策略**:

| 缓存级别 | 淘汰策略 | 触发条件 |
|----------|----------|----------|
| L1 | LRU | 超过 1000 条目 |
| L2 | LFU + TTL | 内存超过 80% |
| L3 | 用户主动删除 | - |

### Q4: 向量数据库选型对比 (Milvus/Pinecone/Chroma)，淘天场景怎么选？

**分类**: 技术选型/向量数据库
**标签**: #向量数据库 #Milvus #RAG #技术选型

**参考答案**:

#### 三大向量数据库对比

| 维度 | Milvus | Pinecone | Chroma |
|------|--------|----------|--------|
| **类型** | 开源 | SaaS 服务 | 开源 |
| **部署方式** | 自建/云托管 | 仅 SaaS | 本地/自建 |
| **规模** | 十亿级 | 十亿级 | 百万级 |
| **查询延迟** | ~10ms | ~5ms | ~20ms |
| **索引类型** | IVF、HNSW、DiskANN | 自研 | HNSW |
| **多租户** | ✅ 强 | ✅ 强 | ⚠️ 弱 |
| **数据持久化** | ✅ | ✅ | ⚠️ 有限 |
| **混合检索** | ✅ (标量 + 向量) | ✅ | ⚠️ 基础 |
| **成本** | 中 (自建) | 高 (按量) | 低 |
| **运维复杂度** | 高 | 低 | 中 |

#### Milvus 详解

**优势**:
- 云原生架构，支持 Kubernetes 部署
- 丰富的索引类型 (IVF_FLAT, HNSW, DiskANN)
- 支持混合检索 (向量 + 标量过滤)
- 活跃的开源社区 (Zilliz 支持)

**劣势**:
- 架构复杂 (Coordinator、DataNode、QueryNode 等)
- 运维成本高
- 学习曲线陡峭

**适用场景**: 大规模、高并发、需要自建

#### Pinecone 详解

**优势**:
- 完全托管，零运维
- 极低延迟 (<5ms)
- 自动扩缩容
- 优秀的开发者体验

**劣势**:
- 仅 SaaS，数据出境问题
- 成本高 (按查询量计费)
- 定制能力有限

**适用场景**: 快速上线、海外业务、预算充足

#### Chroma 详解

**优势**:
- 极简 API，上手快
- 轻量级，可嵌入应用
- 开源免费

**劣势**:
- 规模有限 (百万级)
- 功能相对简单
- 生产级特性不足

**适用场景**: 原型开发、小规模应用、本地部署

#### 淘天场景选型建议

**推荐：Milvus (自建) + 备用方案**

**理由**:

1. **数据合规**: 电商数据涉及用户隐私，不能出境 → 排除 Pinecone
2. **规模需求**: 商品库、用户行为数据达十亿级 → Chroma 不够用
3. **混合检索**: 需要"向量相似度 + 类目/价格/品牌"过滤 → Milvus 强项
4. **成本可控**: 自建长期成本低于 SaaS
5. **技术积累**: 阿里内部有向量检索技术积累

**架构建议**:
```
┌─────────────────────────────────────────┐
│          Milvus 集群 (主)                │
│  • 3 节点 QueryNode                      │
│  • 2 节点 DataNode                       │
│  • 1 节点 Coordinator                    │
│  • 索引：HNSW (M=16, efConstruction=200)│
└─────────────────────────────────────────┘
              │
              ├──────────────┐
              ▼              ▼
    ┌─────────────┐  ┌─────────────┐
    │  Redis 缓存  │  │  ES 兜底    │
    │  热点向量   │  │  全文检索   │
    └─────────────┘  └─────────────┘
```

**优化策略**:
1. **分库分表**: 按商品类目/用户 ID 分片
2. **增量索引**: 新商品实时插入，定时合并
3. **查询路由**: 根据 query 类型路由到不同索引
4. **降级方案**: Milvus 故障时降级到 ES 关键词检索

class FunctionCallTracker:
    def __init__(self, max_same_tool_calls: int = 3):
        self.call_history = []  # [(tool_name, timestamp), ...]
        self.max_same_tool_calls = max_same_tool_calls
    
    def check_loop(self, tool_name: str) -> bool:
        now = time.time()
        recent_calls = [
            name for name, ts in self.call_history
            if now - ts < 60  # 最近 1 分钟
        ]
        
        # 检查是否重复调用同一工具
        if recent_calls.count(tool_name) >= self.max_same_tool_calls:
            return True  # 检测到循环
        
        self.call_history.append((tool_name, now))
        # 清理历史记录
        self.call_history = [
            (n, t) for n, t in self.call_history
            if now - t < 300  # 保留 5 分钟
        ]
        return False

class CircuitBreaker:
    def __init__(self, failure_threshold: int = 5, recovery_time: int = 60):
        self.failure_threshold = failure_threshold
        self.recovery_time = recovery_time
        self.failures = 0
        self.last_failure_time = 0
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN
    
    def call(self, func: Callable, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_time:
                self.state = "HALF_OPEN"
            else:
                raise CircuitOpenError("熔断器打开")
        
        try:
            result = func(*args, **kwargs)
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"
                self.failures = 0
            return result
        except Exception as e:
            self.failures += 1
            self.last_failure_time = time.time()
            if self.failures >= self.failure_threshold:
                self.state = "OPEN"
            raise

# 工具注册时包装

#### 问题根源

- Temperature > 0 引入随机性

- 模型版本更新

async def deterministic_generate(prompt: str, schema: dict, max_retries: int = 3):
    last_error = None
    
    for attempt in range(max_retries):
        response = await llm.generate(
            prompt,
            temperature=0,
            response_format=schema
        )

```python
class LLMCache:
    def __init__(self):
        self.cache = Redis()
    
    def _cache_key(self, prompt: str, config: dict) -> str:
        # 使用 prompt + 配置的哈希作为 key
        content = f"{prompt}|{json.dumps(config, sort_keys=True)}"
        return f"llm:cache:{hashlib.sha256(content.encode()).hexdigest()}"
    
    async def generate(self, prompt: str, config: dict) -> str:
        key = self._cache_key(prompt, config)
        
        # 检查缓存
        if cached := await self.cache.get(key):
            return cached
        
        # 生成
        response = await llm.generate(prompt, **config)
        normalized = normalize_llm_output(response)
        
        # 缓存 (24 小时)
        await self.cache.set(key, normalized, ex=86400)
        
        return normalized
```

### Q7: 如何设计 Agent 的评测体系？如何量化 Agent 的效果？

**分类**: 工程实践/评测
**标签**: #Agent #评测 #指标 #质量保障

**参考答案**:

#### 评测维度框架

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 评测体系                            │
├─────────────────────────────────────────────────────────────┤
│  1. 任务完成度 (Task Completion)                            │
│     • 成功率  • 完成率  • 准确率                             │
├─────────────────────────────────────────────────────────────┤
│  2. 交互质量 (Interaction Quality)                          │
│     • 响应延迟  • 多轮对话连贯性  • 用户满意度               │
├─────────────────────────────────────────────────────────────┤
│  3. 工具使用 (Tool Usage)                                   │
│     • 工具选择准确率  • 调用成功率  • 参数正确率             │
├─────────────────────────────────────────────────────────────┤
│  4. 安全合规 (Safety & Compliance)                          │
│     • 敏感内容过滤  • 权限控制  • 数据泄露检测               │
├─────────────────────────────────────────────────────────────┤
│  5. 成本效率 (Cost & Efficiency)                            │
│     • Token 消耗  • API 调用次数  • 平均处理时间             │
└─────────────────────────────────────────────────────────────┘
```

#### 核心指标定义

**1. 任务完成率 (Task Success Rate, TSR)**

```
TSR = 成功完成的任务数 / 总任务数 × 100%

细分:
- 一次性成功率：无需修正/重试
- 最终成功率：包含重试后成功
- 平均重试次数：失败后重试的平均次数
```

**2. 工具选择准确率 (Tool Selection Accuracy)**

```
TSA = 正确选择工具的任务数 / 需要工具的任务数 × 100%

评估方法:
- 人工标注标准答案
- 与规则基线对比
```

**3. 用户满意度 (Customer Satisfaction, CSAT)**

```
CSAT = (满意评价数 / 总评价数) × 100%

收集方式:
- 显式：点赞/点踩、1-5 星评分
- 隐式：对话时长、转人工率、复购率
```

**4. 响应延迟 (Response Latency)**

```
P50 延迟：50% 请求的响应时间
P95 延迟：95% 请求的响应时间
P99 延迟：99% 请求的响应时间

分解:
- LLM 生成时间
- 工具调用时间
- 网络传输时间
```

**5. 成本指标 (Cost Metrics)**

```
单任务成本 = Token 消耗 × 单价 + API 调用成本

优化目标:
- 平均 Token/任务
- 缓存命中率
- 模型路由效率
```

        self.batch_size = batch_size

#### 实现方案

**1. 模糊度检测**

```python
FUZZY_KEYWORDS = [
    "好看", "不错", "随便", "看看", "推荐",
    "适合", "合适", "一般", "普通"
]

def detect_fuzziness(user_input: str) -> float:
    # 基于关键词
    keyword_score = sum(1 for kw in FUZZY_KEYWORDS if kw in user_input)
    
    # 基于意图分类置信度
    intent_result = intent_classifier.predict(user_input)
    confidence_score = 1 - intent_result.confidence
    
    # 基于信息量 (字符数、实体数)
    entities = extract_entities(user_input)
    info_score = 1 - (len(entities) / 3)  # 期望至少 3 个实体
    
    fuzziness = (keyword_score * 0.4 + confidence_score * 0.4 + info_score * 0.2)
    return min(fuzziness, 1.0)

# 使用
fuzziness = detect_fuzziness("我想买点好看的")
if fuzziness > 0.6:
    strategy = "clarify_and_explore"
```

#### 对话状态追踪

```python
class DialogueState:
    def __init__(self, user_id: str):
        self.user_id = user_id
        self.slot_values = {}  # 已填充的槽位
        self.history = []  # 对话历史
        self.intent_confidence = 0.0
    
    def update(self, user_input: str, extracted_slots: dict):
        self.slot_values.update(extracted_slots)
        self.history.append({"role": "user", "content": user_input})
        
        # 检查是否足够清晰
        required_slots = ['category', 'price_range']
        self.intent_confidence = len([
            s for s in required_slots if s in self.slot_values
        ]) / len(required_slots)
    
    def is_clear(self) -> bool:
        return self.intent_confidence >= 0.8
```

### Q10: 用户说"这个太贵了"，Agent 应该如何理解和回应？

**分类**: 场景/价格敏感处理
**标签**: #电商 #价格敏感 #用户心理

**参考答案**:

#### 语义理解

"这个太贵了" 可能的含义：
1. **超出预算**: 用户有心理价位，商品超出预期
2. **价值不匹配**: 觉得商品不值这个价
3. **比价行为**: 在其他平台看到更便宜的
4. **议价试探**: 希望有优惠/折扣
5. **委婉拒绝**: 其实不感兴趣，用价格当借口

async def handle_price_objection(user_id: str, item_id: str, context: dict) -> Response:
    # 获取商品信息
    item = await get_item(item_id)
    
    # 获取用户预算 (历史行为推断)
    user_budget = await infer_user_budget(user_id)
    
    # 判断价格敏感度
    price_gap = (item.price - user_budget) / user_budget
    
    if price_gap > 0.5:
        # 差距太大，推荐低价替代
        alternatives = await find_similar_items(
            category=item.category,
            price_range=(user_budget * 0.8, user_budget * 1.2)
        )
        return Response(
            text=f"理解您的考虑。我找了几款相似的，价格更实惠：",
            items=alternatives[:3]
        )
    
    elif price_gap > 0.2:
        # 差距适中，强调价值 + 优惠
        coupons = await get_available_coupons(user_id, item_id)
        value_points = await get_item_value_points(item_id)
        
        return Response(
            text=f"理解价格考虑。这款{value_points[0]}，而且现在有优惠：",
            coupons=coupons,
            action="apply_coupon"
        )
    
    else:
        # 差距小，直接给优惠
        return Response(
            text="给您申请了一张专属优惠券：",
            coupon={"discount": 20, "threshold": item.price},
            action="buy_now"
        )
